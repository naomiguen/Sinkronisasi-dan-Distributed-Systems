BAB 1 – PENDAHULUAN
================================================================

1.1 Latar Belakang

Dalam sistem terdistribusi modern, koordinasi antar proses yang berjalan di mesin berbeda
merupakan tantangan fundamental yang tidak dapat diabaikan. Tanpa mekanisme sinkronisasi
yang tepat, berbagai masalah klasik dapat muncul kapan saja, mulai dari inkonsistensi data
akibat akses bersamaan, deadlock karena proses saling menunggu satu sama lain, hingga
starvation ketika sebuah proses tidak pernah mendapatkan giliran untuk dieksekusi.

Masalah ini diperparah oleh tidak adanya global clock yang akurat dalam sistem tersebar.
Setiap node memiliki jam lokalnya sendiri yang dapat berbeda satu sama lain — fenomena
yang dikenal sebagai clock skew. Contoh nyata dampaknya dapat dilihat pada skenario
kompilasi terdistribusi: jika komputer editor memiliki jam yang lebih lambat dari komputer
compiler, perintah make tidak akan melakukan recompile meskipun source code sudah diubah,
karena timestamp source code tampak lebih lama dari file objeknya.

Mengacu pada van Steen dan Tanenbaum (distributed-systems.net), algoritma sinkronisasi
dalam sistem terdistribusi harus memenuhi empat sifat utama. Pertama, informasi yang
relevan tersebar di banyak komputer sehingga tidak ada satu titik yang memiliki gambaran
lengkap. Kedua, setiap keputusan dibuat hanya berdasarkan informasi lokal yang dimiliki
node tersebut. Ketiga, kegagalan akibat penyebab tunggal harus dihindari karena dapat
menghentikan seluruh sistem. Keempat, tidak tersedia sumber waktu global yang akurat
sehingga urutan kejadian harus disepakati melalui mekanisme lain seperti logical clock
Lamport.

1.2 Tujuan Penulisan

Laporan ini mendokumentasikan proses perancangan dan implementasi sistem sinkronisasi
terdistribusi yang mencakup empat komponen utama. Secara lebih rinci, tujuan yang ingin
dicapai dalam tugas ini adalah sebagai berikut.

Distributed Lock Manager (25 poin)
Mengimplementasikan mekanisme penguncian sumber daya terdistribusi menggunakan algoritma
Raft Consensus dari scratch, dengan dukungan tipe shared lock dan exclusive lock,
penanganan network partition, serta deteksi deadlock menggunakan algoritma DFS pada
wait-for graph.

Distributed Queue System (20 poin)
Mengimplementasikan antrian terdistribusi menggunakan consistent hashing untuk
mendistribusikan queue ke node-node dalam cluster, dengan jaminan at-least-once delivery,
persistensi pesan ke Redis untuk recovery saat node restart, serta dukungan multiple
producers dan consumers secara bersamaan.

Distributed Cache Coherence (15 poin)
Mengimplementasikan protokol MESI (Modified, Exclusive, Shared, Invalid) untuk memastikan
konsistensi data antar cache node, dengan pilihan kebijakan eviksi LRU maupun LFU,
mekanisme invalidasi cache via Raft log dan Redis pub/sub, serta pengumpulan metrik
performa menggunakan Prometheus.

Containerization (10 poin)
Mengemas seluruh sistem ke dalam Docker container yang dapat dijalankan dengan satu
perintah docker-compose up, lengkap dengan konfigurasi environment via file .env, health
check otomatis, dan dukungan untuk penambahan node secara dinamis.

1.3 Ruang Lingkup

Sistem dibangun menggunakan Python 3.11 dengan asyncio sebagai fondasi concurrency
non-blocking. Library aiohttp digunakan sebagai HTTP server sekaligus HTTP client untuk
komunikasi RPC antar node. Redis 7 berfungsi sebagai shared distributed state yang
menyimpan lock aktif, pesan queue, dan cache entry secara persisten. Prometheus digunakan
untuk mengumpulkan metrik performa dari setiap node dan memungkinkan analisis throughput,
latency, serta scalability sistem secara real-time.

Sistem dijalankan dengan minimum 3 node secara bersamaan untuk mendemonstrasikan perilaku
distributed yang sesungguhnya. Setiap node menjalankan keempat komponen utama — Raft,
Lock Manager, Queue, dan Cache — secara bersamaan dalam satu proses Python dengan event
loop asyncio tunggal.

[Kotak Informasi] Kaitan dengan Materi Kuliah:
Implementasi dalam tugas ini berkaitan langsung dengan Pertemuan 6 tentang Sinkronisasi.
Raft Consensus merupakan evolusi modern dari algoritma mutual exclusion tersebar yang
dibahas dalam materi kuliah. Konsep clock logika Lamport diimplementasikan sebagai Raft

implementasi nyata dari konsistensi cache berbasis clock yang disebutkan dalam materi
kuliah.

BAB 2 – ARSITEKTUR SISTEM
================================================================

Sistem terdiri dari tiga node identik yang masing-masing menjalankan empat komponen utama
secara bersamaan dalam satu proses Python. Semua node berkomunikasi satu sama lain melalui
HTTP/JSON untuk keperluan Raft RPC, dan menggunakan Redis sebagai penyimpanan state
bersama yang dapat diakses oleh seluruh node. Arsitektur ini dirancang berdasarkan prinsip
design for failure — setiap komponen mengasumsikan bahwa node lain bisa crash kapan saja
tanpa pemberitahuan sebelumnya.

Komponen Utama Per Node

Setiap node dalam cluster menjalankan enam komponen yang saling terhubung. Masing-masing
komponen memiliki tanggung jawab yang jelas dan terdefinisi, sehingga perubahan pada satu
komponen tidak berdampak pada komponen lainnya. Berikut adalah penjelasan singkat fungsi
setiap komponen.

RaftNode: Menjalankan algoritma Raft Consensus secara lengkap: Leader Election, Log
Replication, dan Safety. Setiap write command dari client harus melewati Raft log dan
di-commit oleh mayoritas node sebelum dieksekusi ke state machine.

MessagePassing: Menangani komunikasi HTTP RPC antar node untuk dua jenis pesan Raft:
RequestVote saat election dan AppendEntries untuk replikasi log dan heartbeat. Bertindak
sebagai jembatan antara logika Raft yang abstrak dengan implementasi jaringan yang konkret.

FailureDetector: Melakukan health check ke semua peer setiap satu detik melalui HTTP GET
/health. Node yang gagal merespons tiga kali berturut-turut ditandai SUSPECT, dan yang
gagal enam kali berturut-turut dinyatakan DEAD, sehingga lock yang dimilikinya dapat
dilepas secara otomatis.

LockManager: Mengimplementasikan Distributed Lock Manager dengan dukungan shared dan
exclusive lock, wait queue berbasis asyncio, deteksi deadlock menggunakan DFS, TTL expiry
otomatis, dan persistensi lock ke Redis.

QueueNode: Mengimplementasikan Distributed Queue dengan consistent hashing untuk distribusi
queue ke node, at-least-once delivery via redelivery loop, dan recovery pesan dari Redis
saat node restart.

CacheNode: Mengimplementasikan cache coherence protokol MESI dengan pilihan LRU atau LFU
sebagai replacement policy, dual-path invalidation via Raft log dan Redis pub/sub, serta
TTL expiry otomatis untuk cache entry.

Alur Komunikasi Antar Komponen

Ketika sebuah client mengirim write request (misalnya acquire lock), alurnya adalah sebagai
berikut. Request masuk ke base_node.py melalui HTTP handler, kemudian diteruskan ke
komponen yang bersangkutan (LockManager). LockManager memanggil raft.submit_command()
untuk memasukkan perintah ke log. RaftNode kemudian mengirim AppendEntries via
MessagePassing ke semua Follower. Setelah mayoritas Follower membalas ACK, entry dianggap
committed dan apply_command() dipanggil untuk mengeksekusi perubahan state. Baru kemudian
response dikirim kembali ke client.

Pola ini memastikan bahwa setiap perubahan state selalu melewati Raft consensus dan
terdistribusi ke semua node sebelum dianggap berhasil. Read request seperti cache get dan
lock status langsung dibaca dari state lokal tanpa melewati Raft, sehingga latency-nya
jauh lebih rendah.

3.1 Distributed Lock Manager – Algoritma Raft Consensus (25 poin)

Distributed Lock Manager dibangun di atas algoritma Raft Consensus yang diimplementasikan
dari scratch dalam file src/consensus/raft.py. Raft dipilih karena dirancang khusus untuk
mudah dipahami dibandingkan algoritma Paxos, namun tetap memberikan jaminan keamanan
(safety) yang setara. Raft menyelesaikan masalah konsensus dengan memecahnya menjadi tiga
sub-masalah yang relatif independen, yaitu Leader Election, Log Replication, dan Safety.

Leader Election

Setiap node dalam cluster selalu berada dalam salah satu dari tiga peran: Follower,
Candidate, atau Leader. Pada kondisi normal, hanya ada satu Leader dan semua node lainnya
adalah Follower. Leader bertanggung jawab menerima semua request dari client dan
mendistribusikannya ke Follower.

Setiap Follower memiliki election timeout dengan nilai acak antara 150 hingga 300
milidetik. Nilai acak ini penting untuk menghindari situasi di mana semua node memulai
election secara bersamaan (split vote). Jika sebuah Follower tidak menerima heartbeat dari
Leader dalam rentang waktu tersebut — misalnya karena Leader crash — ia akan bertransisi
menjadi Candidate dan memulai election. Candidate menaikkan term-nya, memberikan suara
untuk dirinya sendiri, lalu mengirimkan RequestVote secara paralel ke semua peer.

Node yang menerima RequestVote akan memberikan suara jika dua syarat terpenuhi: term
candidate lebih tinggi atau sama dari term lokal, dan log candidate minimal se-update log
lokal (election restriction). Syarat kedua ini krusial untuk mencegah kandidat yang lognya
ketinggalan menjadi Leader dan kehilangan data yang sudah di-commit. Candidate yang
mendapat suara mayoritas (2 dari 3 node) langsung menjadi Leader dan mengirim heartbeat
untuk memberitahu semua node. Mekanisme ini memiliki kemiripan konsep dengan Bully
Algorithm dari materi kuliah, namun dengan tambahan election restriction yang membuat Raft
lebih aman.

Log Replication

Setiap perintah yang diterima Leader — baik itu lock acquire, queue produce, maupun cache
set — terlebih dahulu disimpan sebagai LogEntry dalam log lokal Leader. Setiap LogEntry
berisi term saat entry dibuat, index posisi dalam log, dan command yang akan dieksekusi.
Entry ini kemudian dikirim ke semua Follower melalui pesan AppendEntries.

Entry baru dianggap committed hanya setelah mayoritas node (minimal 2 dari 3) berhasil
menyimpannya dan membalas ACK. Setelah committed, Leader memperbarui commit_index dan
semua node mengeksekusi entry tersebut ke state machine melalui fungsi apply_command().
Pendekatan ini memastikan bahwa meskipun Leader crash setelah membalas ke client, entry
yang sudah committed tidak akan hilang karena minimal satu Follower lain sudah memilikinya.
Prinsip ini selaras dengan apa yang dijelaskan Kleppmann dalam DDIA: strong consistency
membutuhkan replikasi ke mayoritas node sebelum write dianggap berhasil.

Implementasi Lock Manager

Lock Manager diimplementasikan dalam file src/nodes/lock_manager.py dan bekerja sebagai
state machine yang di-drive oleh Raft log. Setiap perintah LOCK_ACQUIRE atau LOCK_RELEASE
yang masuk ke Raft log akan dieksekusi oleh fungsi apply_command() di semua node secara
berurutan, memastikan semua node selalu memiliki state lock yang identik.

Sistem mendukung dua tipe lock. Shared lock memungkinkan beberapa client memegang lock
pada resource yang sama secara bersamaan, selama tidak ada yang memegang exclusive lock.
Exclusive lock hanya dapat dipegang oleh satu client, tidak ada client lain yang dapat
acquire lock pada resource tersebut selama exclusive lock aktif. Aturan ini
diimplementasikan dalam fungsi _can_acquire() yang dipanggil sebelum setiap percobaan
acquire.

Setiap lock memiliki TTL (Time-To-Live) yang dikonfigurasi melalui environment variable,
dengan nilai default 30 detik dan maksimum 300 detik. Background loop berjalan setiap satu
detik untuk memeriksa lock yang sudah expired dan melepasnya secara otomatis. Hal ini
mencegah lock zombie — lock yang pemiliknya crash tanpa sempat melepasnya — yang dapat
menyebabkan resource terkunci selamanya. Semua lock yang aktif juga disimpan ke Redis
dengan TTL yang sesuai, sehingga dapat dipulihkan saat node restart.

Deadlock Detection

Deadlock terjadi ketika dua atau lebih client saling menunggu lock yang dipegang oleh satu
sama lain, membentuk siklus dalam wait-for graph. Misalnya, client A menunggu lock yang
dipegang client B, sementara client B menunggu lock yang dipegang client A — situasi ini
tidak akan selesai sendiri tanpa intervensi.

Implementasi menggunakan algoritma DFS (Depth-First Search) pada wait-for graph yang
diimplementasikan dalam kelas DeadlockDetector. Setiap kali sebuah client gagal
mendapatkan lock karena ada holder lain, relasi 'client ini menunggu holder-holder
tersebut' ditambahkan ke graph melalui fungsi add_wait(). Sebelum setiap retry, DFS
dijalankan untuk mencari siklus. Jika siklus ditemukan, request yang memicu siklus
langsung dikembalikan dengan status DENIED beserta informasi siklus deadlock yang
terdeteksi, mencegah sistem terjebak dalam kondisi menunggu selamanya.

Penanganan Network Partition

Network partition ditangani secara inheren oleh algoritma Raft tanpa memerlukan logika
tambahan. Ketika Leader terisolasi dan tidak dapat berkomunikasi dengan mayoritas Follower,
ia tidak dapat mencapai commit untuk write request baru karena tidak ada cukup ACK yang
masuk. Sementara itu, partisi yang masih memiliki mayoritas node akan memilih Leader baru
setelah election timeout. Ketika partisi pulih, Leader lama akan melihat term yang lebih
tinggi dari Leader baru dan langsung mundur menjadi Follower, kemudian menyinkronkan log
yang tertinggal melalui AppendEntries.

3.2 Distributed Queue System (20 poin)

Distributed Queue System diimplementasikan dalam file src/nodes/queue_node.py menggunakan
teknik consistent hashing untuk mendistribusikan queue ke node-node dalam cluster. Berbeda
dengan sharding biasa yang menggunakan modulo sederhana, consistent hashing memastikan
bahwa ketika ada node yang masuk atau keluar dari cluster, hanya sebagian kecil queue yang
perlu dimigrasi — bukan seluruhnya.

Consistent Hashing

Implementasi ConsistentHashRing menggunakan hash MD5 untuk memetakan setiap node ke
posisi-posisi pada sebuah ring virtual. Setiap node memiliki 100 virtual node pada ring
untuk memastikan distribusi yang merata. Ketika sebuah queue baru dibuat, nama queue-nya
di-hash dan ditempatkan pada ring — node terdekat searah jarum jam menjadi primary node
untuk queue tersebut.

Dengan replication factor sebesar dua, setiap queue juga disimpan di node berikutnya pada
ring sebagai backup. Ini memastikan bahwa ketika satu node crash, antrian tidak hilang
karena node lain masih memiliki salinannya. Sifat deterministic dari consistent hashing
juga berarti bahwa queue yang sama akan selalu dipetakan ke node yang sama pada konfigurasi
cluster yang sama, tanpa perlu menyimpan peta distribusi secara eksplisit.

At-Least-Once Delivery

Sistem mengimplementasikan jaminan at-least-once delivery melalui mekanisme acknowledgment
dengan deadline. Ketika sebuah consumer mengambil pesan melalui endpoint /queue/consume,
pesan tersebut tidak langsung dihapus dari antrian. Sebaliknya, pesan berpindah ke
dictionary _in_flight dengan batas waktu ACK yang dikonfigurasi (default 30 detik).

Consumer harus memanggil endpoint /queue/ack dengan message_id yang valid sebelum batas
waktu tersebut. Jika ACK tidak datang dalam waktu yang ditentukan, background loop
_redelivery_loop() yang berjalan setiap lima detik akan mengembalikan pesan tersebut ke
depan antrian dengan delivery_count yang sudah dinaikkan. Pesan akan terus dikirim ulang
sampai akhirnya di-acknowledge. Konsekuensinya, consumer harus didesain bersifat idempotent
— mampu memproses pesan yang sama lebih dari satu kali tanpa menimbulkan efek samping yang
tidak diinginkan. Trade-off ini diterima karena at-least-once jauh lebih mudah
diimplementasikan daripada exactly-once, sambil tetap menjamin tidak ada pesan yang hilang.

Message Persistence dan Recovery

Setiap pesan yang masuk melalui QUEUE_PRODUCE command segera disimpan ke Redis dengan key
queue:{nama_queue}:{message_id}. Penyimpanan ini dilakukan saat apply_command() dipanggil,
sehingga semua node dalam cluster menyimpan salinan pesan secara bersamaan setelah Raft
commit. Ketika sebuah node restart setelah crash, fungsi _restore_from_redis() secara
otomatis membaca kembali semua pesan yang tersimpan di Redis dan membangun ulang state
antrian lokal. Konfigurasi Redis menggunakan AOF dengan appendfsync everysec memastikan
bahwa pesan yang sudah tersimpan tidak hilang meskipun Redis sendiri mengalami crash.

3.3 Distributed Cache Coherence – Protokol MESI (15 poin)

Cache coherence diimplementasikan dalam file src/nodes/cache_node.py menggunakan protokol
MESI. Protokol ini berasal dari dunia hardware multiprocessor cache (Papamarcos dan Patel,
1984) dan diadaptasi untuk konteks software distributed cache. Tujuan utamanya adalah
memastikan bahwa semua node dalam cluster selalu memiliki pandangan yang konsisten terhadap
data yang sama, meskipun setiap node memiliki salinan lokalnya sendiri.

State MESI dan Transisinya

Setiap entry dalam cache memiliki salah satu dari empat state berikut. State Modified (M)
menandakan bahwa entry telah dimodifikasi dan berbeda dari nilai yang tersimpan di Redis
— hanya satu cache node yang boleh berada dalam state ini untuk key yang sama. State
Exclusive (E) menandakan bahwa entry konsisten dengan Redis dan hanya cache ini yang
menyimpannya. State Shared (S) menandakan bahwa beberapa cache node menyimpan entry yang
sama dan semuanya konsisten dengan Redis. State Invalid (I) menandakan bahwa entry tidak
valid dan harus di-fetch ulang dari Redis pada akses berikutnya.

Transisi state terjadi secara otomatis. Ketika sebuah key baru di-set, entry mendapat state
Exclusive karena belum ada node lain yang memilikinya. Ketika key yang sudah ada di-update,
state berubah menjadi Modified karena nilai lokal sudah berbeda dari Redis. Ketika
invalidasi terjadi, state berubah menjadi Invalid di semua node yang menyimpan entry
tersebut.

Dual-Path Invalidation

Invalidasi cache menggunakan dua jalur secara bersamaan untuk memastikan konsistensi yang
cepat dan handal. Jalur pertama melalui Raft log: perintah CACHE_INVALIDATE dimasukkan ke
Raft log dan di-commit ke mayoritas node, memastikan bahwa semua node yang berpartisipasi
dalam Raft consensus akan mengeksekusi invalidasi tersebut dalam urutan yang benar.

Jalur kedua melalui Redis pub/sub: setelah Raft commit, Leader mempublikasikan notifikasi
invalidasi ke channel cache:invalidate di Redis. Semua node yang subscribe ke channel ini
(melalui background task _subscribe_invalidations()) akan segera menginvalidasi entry yang
bersangkutan dari cache lokal mereka. Kombinasi dua jalur ini memberikan kecepatan
propagasi yang tinggi sekaligus jaminan konsistensi yang kuat.

Replacement Policy: LRU dan LFU

Sistem menyediakan dua pilihan replacement policy yang dapat dikonfigurasi melalui
environment variable CACHE_REPLACEMENT_POLICY. LRU (Least Recently Used) diimplementasikan
menggunakan OrderedDict Python yang memungkinkan pemindahan entry ke posisi terbaru dalam
O(1) setiap kali diakses. Ketika cache penuh dan entry baru harus ditambahkan, entry yang
paling lama tidak diakses (berada di posisi paling depan OrderedDict) akan dibuang.

LFU (Least Frequently Used) diimplementasikan menggunakan dictionary biasa dengan counter
frekuensi akses per entry. Ketika eviksi diperlukan, entry dengan nilai frekuensi terendah
yang dipilih untuk dibuang. LFU lebih tepat untuk workload di mana popularitas item
relatif stabil dalam jangka panjang, sedangkan LRU lebih tepat untuk workload dengan
temporal locality yang kuat — di mana item yang baru diakses cenderung diakses lagi dalam
waktu dekat.

3.4 Containerization dan Deployment (10 poin)

Seluruh sistem dikemas dalam Docker container menggunakan base image python:3.11-slim untuk
meminimalkan ukuran image. Setiap node berjalan dalam container terpisah dengan konfigurasi
yang unik melalui environment variables. File docker-compose.yml mendefinisikan seluruh
stack yang dibutuhkan sehingga dapat dijalankan hanya dengan satu perintah.

Docker Compose mengatur enam service: tiga node aplikasi (node-1, node-2, node-3), satu
Redis, dan satu Prometheus yang diaktifkan secara opsional menggunakan Docker Compose
profile. Semua service terhubung melalui network internal cluster-net dengan subnet
172.20.0.0/24, memungkinkan komunikasi antar container menggunakan nama service sebagai
hostname. Setiap node memiliki health check yang melakukan HTTP GET ke /health setiap 10
detik, sehingga Docker otomatis merestart node yang tidak merespons.

File .env.example berisi semua parameter konfigurasi yang tersedia, mulai dari identitas
node (NODE_ID, NODE_PORT), konfigurasi cluster (CLUSTER_NODES), parameter Redis, parameter
Raft (election timeout, heartbeat interval), konfigurasi Lock Manager (default TTL, max
retries), hingga konfigurasi Cache (protokol, replacement policy, max size). Operator cukup
menyalin file ini menjadi .env dan menyesuaikan nilainya sesuai kebutuhan.

4.1 Struktur File Proyek

Struktur direktori proyek mengikuti spesifikasi yang diberikan dalam tugas, dengan
penambahan beberapa file konfigurasi pendukung. Setiap direktori memiliki fungsi yang jelas
dan file-file di dalamnya tidak saling bergantung secara siklik.

distributed-sync-system/
  src/
    consensus/
      raft.py            - Raft Consensus (Leader Election, Log Replication)
    communication/
      message_passing.py - HTTP RPC antar node (RequestVote, AppendEntries)
      failure_detector.py- Phi Accrual Failure Detector
    nodes/
      base_node.py       - Entry point, HTTP server, route semua endpoint
      lock_manager.py    - Distributed Lock + Deadlock Detection
      queue_node.py      - Queue + Consistent Hashing + Redelivery
      cache_node.py      - MESI Cache + LRU/LFU + Pub/Sub Invalidation
    utils/
      config.py          - Centralized config dari environment variables

4.4 Troubleshooting

Berikut adalah masalah yang umum ditemui beserta cara mengatasinya. Jika cluster tidak
memilih Leader setelah beberapa detik, periksa apakah ketiga node bisa saling berkomunikasi
— jalankan docker-compose logs untuk melihat error koneksi. Jika lock acquire selalu
timeout, kemungkinan Raft belum mencapai quorum karena satu node mati — cek /status untuk
melihat state setiap node. Jika Redis connection error, pastikan service redis sudah
healthy sebelum node dimulai — docker-compose up redis terlebih dahulu jika diperlukan.
Untuk debugging Raft, set LOG_LEVEL=DEBUG di .env untuk melihat detail setiap
RequestVote, AppendEntries, dan state transition.

BAB 5 – ANALISIS PERFORMA
================================================================

Pengujian performa dilakukan menggunakan dua pendekatan yang saling melengkapi. Pertama,
benchmarking langsung untuk mengukur latency per operasi dalam kondisi beban rendah.
Kedua, load testing menggunakan Locust untuk mengukur throughput dan perilaku sistem di
bawah beban tinggi dengan banyak concurrent user. Hasil dari kedua pengujian ini
dibandingkan antara konfigurasi single-node dan distributed tiga node untuk
mengkuantifikasi trade-off yang terjadi.

5.1 Perbandingan Single Node vs Distributed

5.2 Analisis Overhead Write Operations

Overhead yang signifikan pada operasi write (+400% hingga +900%) adalah konsekuensi
langsung dan yang memang diharapkan dari mekanisme Raft Consensus. Setiap write operation
harus melewati tiga tahap yang tidak ada pada single-node: pertama, Leader menulis entry
ke log lokalnya; kedua, Leader mengirim AppendEntries secara paralel ke dua Follower dan
menunggu minimal satu ACK (untuk mencapai mayoritas dari 3 node); ketiga, setelah commit,
Leader mengeksekusi command ke state machine dan baru mengirim respons ke client.

Sebagian besar overhead berasal dari dua round-trip network: satu dari Leader ke Follower
untuk AppendEntries, dan satu kembali dari Follower ke Leader untuk ACK. Dalam kondisi
jaringan lokal (Docker network), setiap round-trip memerlukan sekitar 1-2 milidetik,
ditambah waktu pemrosesan di Follower dan overhead asyncio. Total overhead sekitar 35-40
milidetik ini adalah harga yang dibayar untuk mendapatkan jaminan konsistensi dan fault
tolerance yang diberikan Raft.

5.3 Analisis Read Operations

Operasi read seperti Cache Get tidak menunjukkan overhead sama sekali (0%) karena tidak
melewati Raft log. Read langsung dilayani dari state lokal masing-masing node tanpa
koordinasi antar node. Ini menghasilkan throughput yang sangat tinggi — sekitar 5000
request per detik untuk cache get — yang merupakan keunggulan utama dari arsitektur
distributed cache ini.

Queue Consume juga menunjukkan overhead yang rendah (+67%) karena hanya mengambil pesan
dari antrian lokal tanpa submit ke Raft. Yang memerlukan Raft hanyalah operasi ACK, namun
ACK biasanya dilakukan secara asynchronous sehingga tidak menghalangi consumer untuk
segera memproses pesan berikutnya. Pola ini selaras dengan prinsip CQRS (Command Query
Responsibility Segregation) yang direkomendasikan Kleppmann untuk sistem yang membutuhkan
read throughput tinggi.

5.4 Hasil Load Testing dengan Locust

Load testing dilakukan menggunakan empat tipe user simulasi yang berjalan secara
bersamaan, masing-masing dengan bobot yang mencerminkan pola penggunaan realistis.
Berikut adalah konfigurasi dan hasil yang diperoleh.

------------------+-------+------------------------------------------+-------------------
LockUser          |   3   | acquire + release lock bersamaan         | ~180 ms
QueueUser         |   2   | produce, consume, dan ack pesan          | ~160 ms
CacheUser         |   5   | 80% read, 15% write, 5% invalidate       | ~8 ms (read) / ~170 ms (write)
HealthCheckUser   |   1   | GET /health dan GET /status              | ~5 ms

Pada pengujian dengan 50 concurrent user dan spawn rate 5 user per detik, sistem berhasil
mempertahankan error rate di bawah 1% selama 60 detik pengujian. Error yang terjadi
umumnya berupa lock timeout ketika banyak client berebut resource yang sama secara
bersamaan, yang merupakan perilaku yang benar dan diharapkan — bukan crash sistem.

5.5 Analisis Scalability

Sistem dirancang untuk mendukung penambahan node secara dinamis. Raft dapat menangani
hingga f node gagal selama total node adalah 2f+1. Dengan 3 node, sistem toleran terhadap
1 node gagal. Dengan 5 node, toleran terhadap 2 node gagal. Penambahan node meningkatkan
fault tolerance namun juga meningkatkan latency write sedikit karena Leader perlu menunggu
lebih banyak ACK.

Untuk Queue System, consistent hashing memastikan bahwa penambahan node ke-4 hanya
menyebabkan redistribusi sekitar 25% queue (yaitu yang sebelumnya dipetakan ke node baru
tersebut), bukan semua queue seperti yang terjadi pada sharding biasa dengan modulo. Ini
membuat operasi scale-out jauh lebih efisien dan tidak mengganggu queue yang tidak
terpengaruh redistribusi.

BAB 6 – KESIMPULAN DAN REFLEKSI
================================================================

6.1 Pencapaian

Tugas ini berhasil mengimplementasikan sistem sinkronisasi terdistribusi yang mencakup
seluruh core requirements. Berikut adalah ringkasan pencapaian per komponen beserta
penilaian terhadap kelengkapan implementasinya.

Raft Consensus — Lock Manager (25 poin):
Implementasi lengkap mencakup Leader Election dengan election timeout acak, Log Replication
dengan AppendEntries dan commit via mayoritas, Safety melalui election restriction,
recovery node setelah crash, deadlock detection DFS, TTL expiry otomatis, shared dan
exclusive lock, serta persistensi ke Redis.

Distributed Queue System (20 poin):
Consistent hashing dengan 100 virtual node per server, replication factor 2 untuk fault
tolerance, at-least-once delivery via redelivery loop dengan ack_deadline, message
persistence dan recovery dari Redis, serta dukungan multiple producers dan consumers
secara bersamaan.

Cache Coherence MESI (15 poin):
Implementasi lengkap empat state MESI dengan transisi yang benar, dual-path invalidation
via Raft log dan Redis pub/sub, pilihan LRU menggunakan OrderedDict dan LFU menggunakan
frequency counter, TTL expiry, dan Prometheus metrics untuk hit rate, miss rate, dan
ukuran cache.

Containerization (10 poin):
Docker Compose dengan 5 service (3 node + Redis + Prometheus opsional), health check pada
setiap node, environment configuration via .env, Docker network internal, volume
persistence untuk Redis, dan support penambahan node secara dinamis.

6.2 Tantangan dalam Implementasi

Beberapa tantangan signifikan ditemui selama proses implementasi. Memahami dan
mengatasinya menjadi pembelajaran paling berharga dari tugas ini.

Raft Election Timing: Election timeout yang terlalu kecil menyebabkan split vote berulang
di mana beberapa node memulai election secara hampir bersamaan dan tidak ada yang mencapai
mayoritas. Solusinya adalah menggunakan range acak 150-300ms yang memberikan cukup jarak
antar node untuk memulai election secara bergantian.

Asyncio Concurrency: Semua operasi I/O harus benar-benar non-blocking agar event loop
tidak tersumbat. Kesalahan umum adalah memanggil fungsi synchronous di dalam coroutine.
Penggunaan asyncio.gather() untuk mengirim AppendEntries ke semua peer secara paralel
sangat krusial untuk performa heartbeat yang tepat waktu.

Deadlock Detection Timing: Deteksi deadlock harus dijalankan sebelum setiap retry, bukan
hanya sekali di awal. Siklus dalam wait-for graph bisa terbentuk secara bertahap — A
menunggu B, kemudian B mulai menunggu A — sehingga deteksi yang hanya sekali tidak akan
menangkap kasus ini.

Redis Pub/Sub dan Raft: Menggunakan dua jalur untuk cache invalidation memerlukan desain
yang hati-hati agar tidak terjadi double-invalidation atau race condition. Solusinya adalah
memastikan Redis pub/sub hanya digunakan sebagai notifikasi tambahan, bukan sebagai
mekanisme consensus utama.

6.3 Kaitan dengan Materi Perkuliahan

Implementasi ini memberikan pemahaman yang jauh lebih dalam terhadap konsep-konsep yang
dibahas dalam Pertemuan 6 tentang Sinkronisasi. Algoritma Raft merupakan implementasi
modern dari prinsip algoritma tersebar (distributed mutual exclusion) yang dibahas dalam
materi kuliah, dengan tambahan mekanisme Log Replication yang tidak ada dalam algoritma
Ricart-Agrawala klasik. Leader Election dalam Raft memiliki analogi langsung dengan Bully
Algorithm — keduanya memilih node dengan identitas tertinggi sebagai koordinator, namun
Raft menggunakan term sebagai pembanding dan menambahkan election restriction untuk safety.

Clock logika Lamport yang dibahas dalam materi kuliah diimplementasikan secara implisit
melalui Raft term: setiap node hanya menerima pesan dari term yang sama atau lebih tinggi,
memastikan urutan kausalitas yang konsisten di seluruh cluster. MESI Protocol merupakan
implementasi nyata dari konsep konsistensi cache berbasis clock yang disebutkan dalam
materi, di mana state entry mencerminkan hubungan kausalitas antara nilai lokal dan nilai
di penyimpanan terpusat (Redis).
